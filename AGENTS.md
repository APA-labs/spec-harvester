# Working Rules

## 사용자 요청 기억/기록 원칙
- 사용자가 "이후 작업에도 기억되어야 하는 규칙"을 요청하면, 해당 내용을 이 문서에 즉시 추가/갱신한다.
- 규칙은 실행 가능한 체크리스트 형태로 간결하게 작성한다.
- 기존 규칙과 충돌 시, 최신 사용자 지시를 우선 반영하고 변경 이력을 명확히 남긴다.

## 기본 Git 작업 순서 (매 작업)
1. `main` 최신 상태를 기준으로 새 브랜치를 만든다. 브랜치 접두사는 `feature/`를 사용한다.
2. 작업 구현 후 변경 파일을 점검한다.
3. 커밋한다.
4. PR을 생성한다.
5. 가능하면 에이전트가 직접 `main` 병합까지 진행한다.
6. 작업 종료 시 로컬 브랜치를 `main`으로 전환한다.

## 병합/권한 관련 원칙
- 병합 권한, 보호 브랜치 규칙, CI 실패 등으로 자동 병합이 불가능하면 즉시 사유를 공유하고 다음 최소 액션(재시도 명령 또는 사용자 승인 필요 항목)을 제시한다.
- 파괴적 명령은 사용자의 명시적 요청 없이 실행하지 않는다.

## 커뮤니케이션 원칙
- 각 작업 시작 시, 이번 작업에서 적용할 브랜치/커밋/PR/병합/메인복귀 순서를 짧게 다시 확인한다.
- 작업 완료 보고 시 아래 항목을 포함한다.
  - 생성 브랜치명
  - 커밋 해시/메시지
  - PR 링크
  - 병합 여부
  - 최종 체크아웃 브랜치(`main`)
